- 동일 정의가 여러 파일에 있으면 Spine에는 **최종 정의 1개만**.

## 3. 교차 검증(핀포인트) 인터페이스 명세

### 3-1) R1B ↔ R3(사회/계층)
- 검증 질문
  - ICX/JSA 결과가 **신분/계층/거주/배급**에 “어떤 법적 링크”로 연결되는가?
  - R1B에 ‘연동 근거 조항(링크)’만 존재하는가? (정책 설계가 R1B로 역류하지 않는가?)

### 3-2) R1C/R1D2 ↔ R4(과학/기술)
- 검증 질문
  - R1에서 언급한 장비/금기기술이 R4 원장 스펙과 모순 없는가?
  - R1에는 ‘운용/규제’만 있고 ‘스펙/공식’이 남아있지 않은가?

### 3-3) R1D1 ↔ R2(역사/사건)
- 검증 질문
  - 금기기술 통제와 전시 즉결 절차가 특정 역사 사건과 시간·인과 모순을 내지 않는가?

## 4. 정본(Real Lore) 내려쓰기 프로토콜

### 4-1) 내려쓰기 규칙
- 정본에는 **확정된 법령/규칙/스펙 선언문**만.
- 반대의견/비판/대안/레드팀은 **정본에 절대 포함 금지**.
- 정본 문장은 ‘판결문/시행령’ 톤으로.

### 4-2) 내려쓰기 순서
1) Spine의 최종 정의 확정
2) A→B→C→D 순으로 ‘규범 스펙’ 확정
3) 교차 검증 완료 표시(체크)
4) 정본으로 추출(복사)하되, 각 문장에는 원 소스 링크/근거 링크를 메타로만 남김

---

## 5. 실행 체크리스트(작업자가 그대로 따라하면 되는 단계)

1) `R1_Spine_...` 생성 후 링크 뼈대/Issue Queue 틀 만들기
2) A/A2/B/C/D1/D2 파일 생성 후 “빈 목차”만 먼저 세팅
3) 각 입력 소스 파일에서 **결론 문장만** 발췌해 해당 서브마스터에 편입
4) 중복 정의는 ‘공통 규칙’ 섹션으로 이동
5) 기술 스펙/계산식/알고리즘은 즉시 R4 이관 표시
6) 교차 검증 질문 3세트에 대해 ‘모순 없음/해결 필요’로 표기
7) 최종 확정된 문장만 모아 `R1_Lore_정본_...` 생성
